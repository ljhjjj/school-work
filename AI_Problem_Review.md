# agent_rules 종합 문제점 분석

> 분석 대상: `agent_rules/` 폴터 전체 (8개 Markdown + 3개 JSON, 총 1,300줄 이상)
> 분석 일시: 2026-05-01
> AI 에이전트 행동 제어 규칙의 구조적, 운영적, 기술적 문제를 종합 정리

---

## 1. 복잡도 폭발 (Cognitive Overload)

**문제 요약**
- 총 11개 파일, 1,300줄 이상의 규칙으로 AI가 매 작업마다 완벽히 준수하기 현실적으로 불가능합니다.
- `System Prompt`에서 "같은 가이드를 불필요하게 반복 읽지 마라"고 경고하는 것 자체가 이미 규칙이 너무 많아 반복 읽기로 인한 실행 지연이 문제가 되었음을 보여줍니다.
- 9단계 표준 워크플로우(Guide E → F → A → B → C → D → G → A → E)는 이론적으로는 체계적이나, 실제 AI 컨텍스트 한계상 100% 준수가 어렵습니다.

**영향도**
- AI가 핵심 규칙과 부수 규칙을 구분하지 못하고 중요한 안전 정책을 놓칠 수 있습니다.
- 작업 시작 전 가이드 읽기만으로도 상당한 토큰 소모가 발생합니다.

**개선 방향**
- Tier 기반 규칙 축소: 필수(P0), 권장(P1), 참고(P2)로 우선순위를 명시적으로 분리합니다.
- 워크플로우 단순화: 9단계를 5단계(준비 → 입력 → 작성 → 검증 → 마무리)로 축약합니다.

## 2. JSON과 Markdown의 이중성 및 중복

**문제 요약**
- 동일한 내용이 Markdown 서술과 JSON 스키마에 중복/분산되어 있습니다.
  - `dangerous_commands`: `codex_execution_guide.md` 5.2절 + `codex_policy.json`
  - `report_templates`: `codex_execution_guide.md` 8.2절 + `report_templates.json`
  - `validation` 체크항목: `validation_guide.md` + `validation_rules.json`
- 유지보수 시 양쪽을 동기화하지 않으면 불일치가 발생할 위험이 큽니다.

**영향도**
- 규칙 변경 시 2개 이상의 파일을 동시에 수정해야 하며, 누락 시 AI가 혼란을 겪습니다.
- JSON 파일 누락 시 fallback으로 "기본 지식"을 사용한다고 하지만, 이는 일관성 없는 동작을 유발합니다.

**개선 방향**
- Single Source of Truth 원칙 적용: 기계적 판단이 필요한 규칙은 JSON에, 설명/맥락은 Markdown에만 남기고 중복을 제거합니다.
- JSON을 canonical source로 삼고, Markdown에서는 "자세한 내용은 `schemas/xxx.json` 참조" 형태로 축약합니다.

## 3. 강한 Windows / 강의노트 도메인 종속

**문제 요약**
- **OS 종속**: PowerShell(`Get-Content`, `Out-File`, `New-Item`), `conda run -n pdfkit310`, Windows 경로 등이 하드코딩되어 있습니다.
- **도메인 종속**: `inbox/과목명/주차-차시/` 구조, 대학 강의노트라는 특수 목적, `lecture_slide.pdf`/`handwritten_note.pdf` 등 고정 파일명 규칙은 다른 프로젝트로의 범용 적용을 차단합니다.

**영향도**
- macOS/Linux 환경에서는 규칙 전체를 재작성해야 합니다.
- 강의노트가 아닌 다른 문서(논문, 보고서, 메뉴얼) 생성에는 재사용이 거의 불가능합니다.

**개선 방향**
- OS 추상화 레이어 도입: `shell_profile` 설정으로 PowerShell/Bash/Zsh를 선택 가능하게 합니다.
- 도메인 설정 분리: `subject_pattern`, `folder_structure` 등을 `schemas/config.json`로 분리하여 강의노트 외 도메인에서도 구조만 교체하면 사용 가능하게 합니다.

## 4. 정규식 검증의 경직성

**문제 요약**
- `validation_rules.json`의 H1 패턴(`^# .+ \d+-\d+: .+$`)은 **"과목명 주차-차시: 주제" 형식만 허용**합니다.
- H2 패턴(`^## \d+\. .+`)은 반드시 숫자로 시작해야 하므로 자유로운 섹션 제목을 허용하지 않습니다.
- 주차/차시가 1자리수를 넘어가거나(예: `10-1`), 주제에 특수문자가 포함되면 regex가 깨질 수 있습니다.

**영향도**
- 유연한 문서 구조를 강제로 규격에 맞춰야 하므로, 다양한 강의 형식(특강, 세미나, 실습)에 대응하지 못합니다.
- 정상적인 마크다운이라도 검증 실패로 처리되어 불필요한 재작업이 발생합니다.

**개선 방향**
- H1/H2 패턴을 더 느슨하게 완화하거나, `schemas/validation_rules.json`에서 도메인별로 오버라이드 가능하게 합니다.
- 예: H1을 `^# .+$`로 완화하고, 선택적 `week_session` 패턴은 별도의 권장 규칙으로 분리합니다.

## 5. LaTeX 매크로 제한의 과도함

**문제 요약**
- `validation_rules.json`의 `discouraged_macros`에 `\underbrace`, `\overbrace`, `\begin{aligned}`, `\begin{array}` 등이 포함되어 있습니다.
- 이들은 공학/수학 강의노트에서 **매우 일반적으로 사용**되는 매크로입니다.
- 특히 `aligned` 환경은 다중 행 수식의 표준입니다.

**영향도**
- 정상적인 수식 표기가 `warning`으로 잡혀 불필요한 잔여 이슈가 증가합니다.
- AI가 의도적으로 수식 품질을 낮춰서 warning을 회피할 수 있습니다.

**개선 방향**
- `discouraged_macros` 목록을 재검토하여 실제로 렌더링 충돌을 일으키는 매크로만 남깁니다.
- `aligned`와 `array`는 `warning`에서 제외하고, `report_only`가 아닌 `allowed`로 변경합니다.

## 6. manifest 상태 전이의 모호한 트리거

**문제 요약**
- manifest 상태 변경이 "사용자가 **명시적으로** 문서 작성 작업을 요청한 경우"에만 가능하다고 하지만, AI가 "명시적 요청"인지 "암묵적 요청"인지를 판단하는 것은 주관적입니다.
- 검토/분석/dry-run/규칙 질문에서 manifest를 read-only로 취급해야 하지만, 이 경계가 모호합니다.

**영향도**
- 의도치 않게 `pending` → `in_progress`로 전환되거나, 반대로 실제 작성 요청임에도 상태가 변경되지 않을 수 있습니다.
- `blocked` 전환 조건이 "이후 전개 전체의 핵심 전제"인지 판단하는 것도 AI의 주관에 크게 의존합니다.

**개선 방향**
- 트리거를 명령어/키워드 기반으로 명확화합니다.
  - `생성해줘`, `작성해줘`, `정리해줘` → 상태 전이 O
  - `검토해줘`, `분석해줘`, `어떻게 생각해?` → 상태 전이 X
- `task_type` 필드를 manifest에 명시적으로 추가하여 `new_note`, `review`, `dry_run` 등으로 구분합니다.

## 7. PLAN 파일 관리의 현실적 부담

**문제 요약**
- 매 차시마다 `output/과목명/과목명_주차-차시_PLAN.md`를 생성하고, 체크박스(`[x]`/`[ ]`)를 수동 업데이트해야 합니다.
- `페이지 판독 기록`, `입력 커버리지 검증`, `⚠️ 실행 이슈`, `잔여 이슈`, `Phase 1/2/3` 등의 섹션을 모두 관리해야 합니다.
- AI의 **자율성을 표방하면서도 실제로는 엄청난 수작업 메타데이터 관리**를 요구하는 모순이 있습니다.

**영향도**
- AI가 본문 작성보다 PLAN 파일 업데이트에 더 많은 토큰을 소모할 수 있습니다.
- 작업 재개 시 PLAN의 체크박스 상태와 실제 파일 내용이 불일치할 위험이 큽니다.

**개선 방향**
- PLAN 파일을 선택적(optional)으로 전환합니다. 간단한 작업(1~2페이지 분량)은 PLAN 없이 바로 본문 작성을 허용합니다.
- PLAN 템플릿을 간소화하여 `페이지 판독 기록` 등 상세 기록은 선택적으로 남깁니다.
- 체크박스 대신 완료된 섹션 목록만 간단히 기록하는 방식으로 부담을 줄입니다.

## 최종 의견

`agent_rules`는 "대학 강의노트 자동 생성"이라는 매우 구체적인 목적에 대해 **엔터프라이즈급 거버넌스 프레임워크**를 구축했습니다. 하지만 그만큼 **규칙의 양, 복잡도, 도메인 종속성**이 극에 달해 있습니다.

현재 구조는 **처음부터 다시 설계하는 리팩토링** 없이는 장기적인 유지보수가 어려울 것으로 보입니다. 특히 JSON/Markdown 이중성과 1,300줄이 넘는 규칙 양은 AI의 실제 실행 품질을 저해하는 가장 큰 병목입니다.

**우선적으로 해결해야 할 항목**은 다음과 같습니다:
1. JSON을 canonical source로 삼고 Markdown의 중복을 제거
2. Tier 기반 규칙 우선순위 도입으로 핵심 규칙만 강제
3. `validation_rules.json`의 regex 패턴 완화
