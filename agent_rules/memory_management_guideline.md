# Markdown Editorial Agent: Task & Memory Management Guidelines

## 1. 일급 객체화 원칙 (Core Principle)
당신(AI 에이전트)은 방대한 양의 강의 자료(.pdf, .txt 등)를 처리할 때 단기 기억력(Context Window)의 한계를 극복하기 위해, 프로젝트 루트에 있는 `DOC_PLAN.md` 파일을 당신의 '외부 기억 장치(External Memory)'로 사용해야 합니다. 문서를 생성하거나 편집하기 전에 **반드시** 이 파일을 먼저 읽고, 한 파트의 작성이 끝나면 **반드시** 상태를 업데이트하세요.

## 2. 작업 루틴 (Execution Loop)
새로운 문서 정리 요구사항을 받거나 작업을 재개할 때 다음 프로세스를 엄격히 따르세요:

1. **READ & INIT (읽기 및 초기화):** 
   - 가장 먼저 `cat DOC_PLAN.md`를 실행하여 현재 진행 상황과 이전 컨텍스트를 파악합니다. (`DOC_PLAN.md`가 없다면 즉시 생성합니다.)
   - **[초기화 규칙]:** 만약 사용자의 지시가 이전 작업의 '계속(이어쓰기)'이 아니라 **완전히 새로운 과목이나 새로운 주차의 작업**을 시작하는 것이라면, 기존 `DOC_PLAN.md`의 내용을 깔끔하게 모두 지우고 새로운 목표와 목차로 완전히 덮어씌우세요(Overwrite).
2. **THINK:** `# Input_Guide`(또는 `cat agent_rules/input_guide.md`)를 읽고, 명시된 파일 처리 규칙 및 정보 우선순위 병합 지침을 철저히 실행합니다. 병합 및 분석된 내용을 바탕으로 전체 마크다운 문서의 대주제(H2) 단위로 단계를 쪼개어 `DOC_PLAN.md`에 구체적인 목차와 작업 계획을 추가합니다.
3. **ACT:** `DOC_PLAN.md`에 명시된 현재 파트의 작업 목표를 확인한 뒤, `cat agent_rules/note_creation_guide.md`를 실행하여 문서 구조 및 서식 가이드라인을 읽어옵니다. 반드시 이 두 가지(현재 Plan과 서식 규칙)를 완벽하게 준수하여 해당 파트의 마크다운 본문을 작성합니다.
4. **UPDATE (상태 업데이트 및 롤링 압축):** 
   - 하나의 소주제(H3) 작성이 완료되면 즉시 `DOC_PLAN.md`의 체크박스를 `[x]` 처리하고 메모를 남깁니다.
   - **[롤링 압축 규칙]:** 단일 문서 작업 중 특정 Phase(예: Phase 1)의 모든 하위 작업이 끝났다면, 길어진 세부 체크리스트는 모두 지우고 `> ✅ Phase 1: 원본 데이터 분석 완료` 형태로 한 줄 요약하여 문서를 압축하세요. 단, 다음 작업에 필수적인 핵심 `Context Memo`는 남겨두어야 합니다.

5. **TROUBLESHOOT (트러블슈팅 및 예외 탈출구):** 
   - 원본 데이터의 심각한 누락(스캔 불량, 수식 식별 불가 등) 발생 시, `DOC_PLAN.md`에 이슈를 기록하고 사용자에게 선명한 이미지(.png)나 텍스트 설명을 요청한 뒤 대기하세요.
   - **[Escape Hatch 규칙]:** 동일한 부분에 대한 재요청은 **최대 2회**로 제한합니다. 2회 시도 후에도 사용자의 응답이 없거나 판독이 불가능하면 무한 루프에 빠지지 말고, 해당 위치에 `> ⚠️ [수식/내용 판독 불가: 수동 입력 필요]` 태그를 눈에 띄게 남긴 뒤 **강제로 다음 파트 작성을 이어나가세요.** 
6. **REVIEW & VALIDATE (웹 검증 및 피드백 대기):** 
   - 한 과목의 챕터 작성이 모두 끝났다면 즉시 파일을 병합하지 말고 인터넷 검색(Fact-Check)을 수행하세요.
   - **[고유 표기법 보존 원칙]:** 웹 검색 결과는 STT의 치명적인 오타나 엉뚱한 개념을 바로잡는 용도로만 사용하세요. 교수님 특유의 단순화된 가정이나 독자적인 수식 표기법(Notation)이 원본(PDF 필기 등)에 명확히 존재한다면, **웹 검색 결과보다 원본 강의 자료의 표기법을 무조건 최우선으로 유지해야 합니다.** 인터넷의 일반적인 공식으로 원본을 임의 훼손하지 마세요.
   - 비교 완료 후 [발견된 차이점 / 수정 제안사항]을 요약하여 사용자에게 보고하고 "제안사항을 반영하여 최종 저장할까요?"라고 질문한 뒤 대기하세요.

#### Phase 2: 본문 마크다운 변환 및 서식 적용
- [x] 1. 정상 상태 정상 유동 (SSSF) 개요 작성
- [x] 2. 개별 공학 장치들의 개요 작성
- [ ] 3. 노즐 및 디퓨저 (Nozzles and Diffusers) 파트 👈 **(현재 포커스)**

- [ ] 4. 압축기 및 펌프 파트 👈 **(Blocked)**
  - *Issue:* STT 파일에서 펌프의 위치 에너지 변화 관련 설명 구간이 통째로 누락됨. 사용자에게 해당 부분의 추가 설명이나 캡처본 요청 대기 중.
#### Phase 3: 웹 검색 기반 팩트 체크 및 최종 검수 (Web Research & QA)
- [ ] 1단계: 작성 완료된 마크다운 내용 중 핵심 개념, 공식, 이론을 인터넷 검색을 통해 교차 검증(Fact-Check)
- [ ] 2단계: 검색된 검증 자료와 현재 작성본을 비교 분석하여 오류 의심 부분, 누락된 핵심 설명, 또는 추가 제안사항 리스트업
- [ ] 3단계: 분석 결과를 바탕으로 사용자에게 "비교 및 수정 제안 리포트"를 출력하고, 수정 반영 여부를 질문한 뒤 대기 👈 **(Confirm 대기)**
- [ ] 4단계: 사용자의 피드백(승인/거절/수정)을 반영하여 최종 `.md` 파일 병합 및 저장 완료

## 4. 컨텍스트 메모(Context Memo)의 의무화
단순히 체크리스트만 지우는 것이 아니라, 중요한 문서화 결정사항(예: 특정 전문 용어의 번역 통일 기준, 복잡한 수식의 대체 표기법, STT 오류 교정 내역 등)을 `*Context Memo:*` 항목으로 반드시 기록하세요. 대화가 초기화되어 당신이 이전 작성본을 잊어버리더라도, 이 메모만 읽으면 즉시 문서의 일관된 톤앤매너와 서식을 복구할 수 있어야 합니다.
