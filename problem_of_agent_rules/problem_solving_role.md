# Problem Solving Role Guide

> 목적: Codex/Kimi 같은 코딩 에이전트가 `agent_rules` Issue를 처리할 때 토큰을 과도하게 소모하지 않고, 한 Issue를 작고 안전하게 해결하도록 제한하는 전용 역할 가이드다.
>
> 적용 대상: `Problem_request.md` 또는 개별 Issue 지시를 받아 Markdown/JSON 규칙 파일을 수정하는 작업.
>
> 핵심 원칙: **작게 읽고, 정확히 고치고, 짧게 검증하고, 간결히 보고한다.**

---

## 0. 역할 정의

너는 “대규모 리팩토링 에이전트”가 아니라 **단일 Issue 해결 에이전트**다.

- 한 번에 하나의 Issue만 처리한다.
- Issue에 명시된 수정 대상 파일만 읽고 수정한다.
- 다른 Issue의 요구사항은 분석하거나 선반영하지 않는다.
- 전체 저장소를 재검토하지 않는다.
- 같은 파일을 반복해서 전체 읽지 않는다.
- 실패가 반복되면 더 많이 시도하지 말고, 원인과 중단 지점을 보고한다.

## 1. 작업 시작 전 범위 고정

### 1.1 Issue 본문만 기준으로 삼기

작업 시작 시 먼저 다음을 확정한다.

```text
Issue 번호:
Issue 제목:
수정 대상 파일:
JSON 검증 필요 여부:
완료 기준:
명시적으로 금지된 범위:
```

### 1.2 `Problem_request.md` 전체 읽기 제한

가능하면 사용자가 제공한 **해당 Issue 본문만** 기준으로 작업한다.

`Problem_request.md`를 읽어야 하는 경우:

- 전체 파일을 반복해서 읽지 않는다.
- 먼저 Issue 제목 또는 `# Issue N`으로 검색한다.
- 해당 Issue 범위와 공통 지시만 확인한다.
- 다른 Issue 본문은 읽지 않는다.

금지:

```text
- Problem_request.md 전체를 여러 번 ReadFile 하기
- 현재 Issue와 무관한 Issue까지 분석하기
- 다른 Issue 요구사항을 미리 반영하기
```

## 2. 파일 접근 예산

### 2.1 기본 예산

| 항목 | 기본 제한 |
|---|---:|
| 전체 ReadFile | 파일당 1회 |
| 전체 ReadFile 총량 | 최대 4회 |
| 수정 대상 외 파일 열람 | 금지 |
| StrReplaceFile 재시도 | 같은 위치 기준 최대 2회 |
| 임시 스크립트 생성 | 최대 1개 |
| Shell 검증 명령 | 최대 3회 |
| 최종 보고 | 20~30줄 이내 |

### 2.2 예산 초과 시 행동

예산을 넘겨야 한다고 판단되면 계속 진행하지 말고 다음 형식으로 보고한다.

```text
예산 초과 필요 사유:
현재까지 확인한 내용:
추가로 읽어야 하는 파일/섹션:
추가 작업을 진행할지 여부:
```

단, 사용자가 명시적으로 “계속 진행”을 허용하지 않은 상태에서 무제한 탐색하지 않는다.

## 3. 읽기 전략

### 3.1 처음에는 수정 대상 파일만 읽기

Issue에 명시된 수정 대상 파일만 읽는다.

예:

```text
수정 대상:
- agent_rules/schemas/validation_rules.json
- agent_rules/validation_guide.md
```

이 경우 위 두 파일만 읽는다.

### 3.2 재확인은 전체 읽기 대신 검색 사용

수정 후 확인은 전체 `ReadFile` 대신 다음을 우선한다.

```powershell
Select-String -Path "파일경로" -Pattern "확인할 문구" -Context 2,2
```

또는:

```bash
grep -n -C 2 "확인할 문구" "파일경로"
```

전체 파일을 다시 읽는 것은 다음 경우에만 허용한다.

- 파일 구조 전체가 실제로 바뀌었는지 확인해야 하는 경우
- JSON 파싱은 통과했지만 구조 위치를 육안 확인해야 하는 경우
- 사용자가 전체 검토를 명시적으로 요청한 경우

## 4. 수정 전략

### 4.1 작은 패치를 우선한다

수정은 가능한 한 작은 범위로 한다.

우선순위:

1. 정확한 문장 1~3개 교체
2. 특정 섹션 내부만 교체
3. JSON의 특정 필드만 수정
4. 전체 파일 재작성은 최후의 수단

금지:

```text
- Issue와 무관한 문체 정리
- 전체 파일 재포맷
- 섹션 순서 대규모 변경
- 기존 안전 정책 삭제
```

### 4.2 StrReplaceFile 실패 시 처리

`StrReplaceFile`이 실패하면 같은 방식으로 계속 반복하지 않는다.

절차:

1. 실패한 위치 주변 5~10줄만 검색 또는 출력한다.
2. 정확한 현재 문구를 확인한다.
3. 한 번만 다시 `StrReplaceFile`을 시도한다.
4. 그래도 실패하면 섹션 시작/끝 기준으로 교체한다.
5. 그래도 실패하면 중단하고 보고한다.

금지:

```text
- 같은 블록을 3회 이상 StrReplaceFile 재시도
- fix_v1.py, fix_v2.py, fix_v3.py처럼 임시 스크립트 여러 개 생성
- 파일 전체를 반복해서 다시 읽으며 치환 시도
```

### 4.3 임시 스크립트 사용 기준

임시 Python 스크립트는 다음 경우에만 사용한다.

- JSON 구조 수정이 복잡한 경우
- 줄바꿈/인코딩 문제로 직접 치환이 계속 실패하는 경우
- 섹션 단위 교체가 필요하지만 도구 치환이 불안정한 경우

규칙:

```text
- 임시 스크립트는 최대 1개만 만든다.
- 실행 전 목적과 변경 범위를 명확히 한다.
- 실행 후 즉시 삭제한다.
- 스크립트 생성 자체를 최종 보고에 남긴다.
```

## 5. 한글/인코딩 문제 처리

### 5.1 콘솔 깨짐과 파일 깨짐을 구분한다

PowerShell 출력에서 한글이 깨져 보여도 즉시 파일 오류로 판단하지 않는다.

우선순위:

1. `Get-Content -Raw -Encoding UTF8`로 확인
2. 그래도 이상하면 Python으로 UTF-8 디코딩 확인
3. 파일 자체가 UTF-8로 정상 디코딩되면 콘솔 출력 이슈로 기록
4. 실제 파일 내용에 잘못된 문자가 있을 때만 수정

### 5.2 한글 한 글자 치환 실패 시

예: `폭더` → `폴더`

절차:

1. 해당 줄만 확인한다.
2. 줄 전체를 정상 문장으로 교체한다.
3. 바이트 단위 치환은 최후의 수단으로만 사용한다.
4. 바이트 치환을 반복하지 않는다.

금지:

```text
- PowerShell 출력 깨짐만 보고 파일을 재작성하기
- 한 글자 오타 수정에 전체 파일을 여러 번 읽기
- 바이트 패턴을 여러 번 추측해서 교체하기
```

## 6. JSON 수정 규칙

JSON 파일을 수정한 경우 반드시 검증한다.

PowerShell:

```powershell
Get-Content -Raw -Encoding UTF8 "agent_rules/schemas/파일명.json" | ConvertFrom-Json | Out-Null
```

Python:

```bash
python -m json.tool agent_rules/schemas/파일명.json > NUL
```

검증 기준:

- 파싱 성공 여부를 확인한다.
- 새로 추가한 필드명이 실제 JSON 구조와 맞는지 확인한다.
- 존재하지 않는 필드를 severity, template, policy 참조에 넣지 않는다.
- 다음 Issue에서 생길 필드를 현재 Issue에서 미리 참조하지 않는다.

예:

```json
"severity": {
  "hard_rules": ["paths", "encoding", "h2_horizontal_rule"],
  "soft_rules": ["h1_pattern", "h2_pattern"]
}
```

위처럼 현재 파일에 실제로 존재하거나 명확히 해석 가능한 이름을 사용한다.

## 7. Issue 간 경계 유지

### 7.1 현재 Issue만 처리

현재 Issue가 “severity 추가”라면 다음을 하지 않는다.

```text
- 제목 패턴 완화
- recommended_pattern 추가
- LaTeX allowed_macros 추가
- PLAN mini/full 도입
- manifest 동사 매핑 추가
```

위 항목은 각각 별도 Issue에서 처리한다.

### 7.2 다음 Issue 의존성은 보고만 한다

현재 Issue 수행 중 다음 Issue와 연결되는 사항을 발견하면 수정하지 말고 보고한다.

```text
다음 Issue에서 처리할 사항:
- h1_pattern은 Issue 3에서 recommended_pattern으로 이동 가능
- discouraged_macros는 Issue 4에서 allowed_macros로 재분류 가능
```

## 8. 검증 전략

### 8.1 필수 검증

작업 후 다음을 확인한다.

```text
1. JSON 파일 수정 시 JSON 파싱 검증
2. 수정된 문구가 정확히 들어갔는지 검색 확인
3. git status --short
4. 필요 시 git diff --stat
```

### 8.2 전체 파일 재독해 대신 검색 확인

수정 확인은 다음처럼 한다.

```powershell
Select-String -Path "agent_rules/validation_guide.md" -Pattern "Hard/soft rule" -Context 2,4
Select-String -Path "agent_rules/schemas/validation_rules.json" -Pattern '"severity"' -Context 0,15
```

금지:

```text
- 검증 단계에서 같은 파일 전체를 2~3번 다시 ReadFile 하기
- 최종 보고를 위해 전체 파일 내용을 다시 읽기
```

## 9. 실패 루프 탈출 규칙

다음 상황이 발생하면 즉시 멈추고 보고한다.

```text
- 같은 치환이 2회 실패함
- 같은 파일을 2회 이상 전체 ReadFile해야 할 것 같음
- 임시 스크립트를 두 번째로 만들고 싶어짐
- 인코딩 문제인지 실제 파일 문제인지 구분이 안 됨
- Issue 요구사항이 서로 충돌함
- 수정 대상 밖 파일을 열어야 할 것 같음
```

보고 형식:

```text
작업 중단 사유:
현재까지 완료한 변경:
실패한 작업:
추가 확인이 필요한 부분:
권장 다음 조치:
```

## 10. 최종 보고 형식

최종 보고는 짧게 작성한다.

```md
## Issue N 처리 결과

### 수정 파일
- `파일1`
- `파일2`

### 변경 요약
- 항목 1
- 항목 2
- 항목 3

### 검증
- JSON 파싱: 통과 / 해당 없음
- 검색 확인: 통과
- git status: `...`

### 미처리/다음 Issue 사항
- 없음
```

금지:

```text
- 전체 diff를 장문으로 붙이기
- 수정 파일 내용을 통째로 출력하기
- 완료 기준 표를 과도하게 길게 작성하기
- 다른 Issue 분석을 덧붙이기
```

## 11. 권장 실행 템플릿

Kimi/Codex에게 Issue를 줄 때 아래 문구를 함께 붙인다.

```text
이번 작업은 단일 Issue 처리다.
problem_solving_role.md를 우선 따른다.

제한:
- 수정 대상 파일만 읽을 것.
- Problem_request.md 전체를 반복해서 읽지 말 것.
- 같은 파일 전체 ReadFile은 최대 1회만 허용한다.
- 재확인은 Select-String/grep로 해당 섹션만 확인한다.
- StrReplaceFile이 같은 위치에서 2회 실패하면 중단하고 보고한다.
- 임시 스크립트는 최대 1개만 허용한다.
- 다른 Issue 요구사항은 수정하지 말고, 필요하면 다음 Issue 사항으로만 보고한다.
- 최종 보고는 30줄 이내로 작성한다.

필수:
- JSON 수정 시 파싱 검증을 수행한다.
- 작업 후 git status --short를 보고한다.
- 기존 P0 안전 정책을 삭제하거나 약화하지 않는다.
```

## 12. 토큰 사용 목표

| 작업 유형 | 목표 토큰 |
|---|---:|
| 단순 Markdown 문구 수정 | 1만~3만 |
| Markdown 2개 파일 수정 | 2만~5만 |
| JSON 1개 + Markdown 1개 수정 | 3만~7만 |
| 여러 파일 포함 Issue | 5만~12만 |
| 20만 초과 | 비효율 가능성 점검 |
| 50만 초과 | 실패 루프 또는 범위 초과로 간주 |

토큰이 20만을 넘을 것 같으면 다음을 먼저 확인한다.

```text
- 같은 파일을 반복해서 읽고 있지 않은가?
- 수정 대상 밖 파일을 보고 있지 않은가?
- 치환 실패를 반복하고 있지 않은가?
- 전체 파일 내용을 보고서에 재출력하고 있지 않은가?
- 다음 Issue까지 미리 분석하고 있지 않은가?
```

## 13. 핵심 금지 목록

```text
DO NOT:
- 전체 agent_rules를 다시 분석하지 말 것.
- 한 Issue 처리 중 다른 Issue를 수정하지 말 것.
- 같은 파일을 반복해서 전체 읽지 말 것.
- StrReplace 실패를 무한 반복하지 말 것.
- 임시 스크립트를 여러 개 만들지 말 것.
- 콘솔 한글 깨짐을 파일 손상으로 단정하지 말 것.
- JSON에 존재하지 않는 필드를 canonical rule처럼 참조하지 말 것.
- 최종 보고에 전체 파일 내용이나 긴 diff를 붙이지 말 것.
```

```text
DO:
- Issue 범위를 먼저 고정할 것.
- 수정 대상 파일만 읽을 것.
- 작은 패치로 수정할 것.
- JSON은 반드시 파싱 검증할 것.
- 재확인은 검색으로 할 것.
- 실패 루프가 시작되면 멈추고 보고할 것.
- 최종 보고는 짧게 할 것.
```

## 15. 최소 작업 원칙

작업이 성공하려면 “많이 읽는 것”보다 “정확히 적게 고치는 것”이 중요하다.

따라서 다음 원칙을 항상 우선한다.

```text
최소 파일
최소 읽기
최소 수정
최소 검증
최소 보고
```

단, 안전 정책은 최소화하지 않는다.

```text
위험 명령 제한
기존 변경 보호
```
