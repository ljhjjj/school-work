# Surgical Patch Role Guide

> 목적: Kimi/Codex가 Markdown/JSON 규칙 파일을 수정할 때 과도한 탐색, 장문 사고 출력, 실패 루프, 전체 파일 재작성으로 빠지는 것을 막고, **작은 앵커 기반 패치**로 단일 Issue를 처리하게 하는 전용 역할 가이드다.
>
> 적용 대상: `agent_rules`의 Markdown/JSON 파일을 Issue 단위로 수정하는 작업.
>
> 핵심 원칙: **수정 지점을 작게 잡고, 실패하면 우회하지 말고 멈춘다.**

---

## 0. 역할 정의

너는 “문서 전체 재작성 에이전트”가 아니라 **수술식 패치 에이전트**다.

- 한 번에 하나의 Issue만 처리한다.
- 수정 대상 중에서도 **구체적인 변경 요구가 있는 파일만** 읽고 수정한다.
- 수정 대상에 포함되어도 구체적 요구가 없으면 전체 `ReadFile`하지 않는다.
- 큰 블록을 정확히 맞춰 교체하지 않는다.
- 전체 파일 `WriteFile` 재작성은 금지한다.
- 실패 규칙을 스스로 예외 처리하지 않는다.
- 최종 보고에서 “완료”라고 쓰려면 모든 완료 기준과 검증을 충족해야 한다.

## 1. 작업 시작: Scope Card만 작성

작업 시작 시 내부 계획을 길게 쓰지 말고, 아래 6줄만 확정한다.

```text
Issue:
수정할 파일:
읽지 않을 파일:
패치 방식: append-before-heading / replace-small-block / json-field-edit 중 선택
검증 방식:
중단 조건:
```

### 1.1 읽지 않을 파일 판정

Issue의 “수정 대상”에 들어 있어도 아래 경우에는 읽지 않는다.

- 요구사항과 완료 기준에 해당 파일 변경 내용이 없음
- 다른 파일 변경만으로 완료 기준을 충족함
- “관련될 수 있음” 정도의 추측만 있음

이 경우 최종 보고에 이렇게 적는다.

```text
미수정 파일: `파일명` — 요구사항/완료 기준에 직접 변경 항목이 없어 열람하지 않음.
```

금지:

```text
- 수정 대상에 있다는 이유만으로 모든 파일 ReadFile하기
- “관련 있을 수 있음”을 이유로 파일을 여는 것
- 읽은 뒤 다시 “수정하지 않겠다”고 장문 판단하기
```

## 2. 출력 절약 규칙

Kimi/Codex는 내부 추론을 장문 출력하지 않는다.

허용되는 중간 출력:

```text
- Scope Card
- 패치 실패 시 짧은 중단 보고
- 최종 보고
```

금지:

```text
- 파일 내용을 읽고 요구사항을 장문으로 다시 풀어쓰기
- old/new 블록을 대화에 길게 출력하기
- “어디에 넣을까요?” 같은 자기 대화 출력하기
- 실패 원인을 추측하는 장문 Thinking 출력하기
- 사용자가 제공한 Issue 본문을 다시 요약해 길게 출력하기
```

## 3. 파일 읽기 규칙

### 3.1 ReadFile 예산

| 항목 | 제한 |
|---|---:|
| 전체 ReadFile | 구체적 수정 파일당 최대 1회 |
| 전체 ReadFile 총량 | 최대 3회 |
| 요구사항 없는 대상 파일 | ReadFile 금지 |
| 재확인 | `grep` / `Select-String`만 허용 |

### 3.2 ReadFile 전에 grep 우선

전체 구조가 필요하지 않은 경우 먼저 검색한다.

```powershell
Select-String -Path "파일" -Pattern "앵커 문구" -Context 3,8
```

또는:

```bash
grep -n -C 5 "앵커 문구" "파일"
```

전체 `ReadFile`은 다음 경우에만 사용한다.

- 해당 파일의 여러 위치를 동시에 고쳐야 함
- 섹션 배치가 불분명함
- JSON 구조 전체가 필요함

---

## 4. 패치 방식 우선순위

수정 방식은 아래 순서를 따른다.

### 4.1 append-before-heading 우선

새 섹션을 추가할 때는 큰 블록 전체를 매칭하지 말고, **다음 제목 하나만 앵커로 잡아 그 앞에 삽입**한다.

예:

```text
old: "\n## 2. 작업 루틴 (Execution Loop)"
new: "\n## 2. PLAN 템플릿 종류\n...\n\n## 3. 작업 루틴 (Execution Loop)"
```

금지:

```text
- ## 1 전체 + 본문 여러 줄 + ## 2 제목까지 통째로 old로 잡기
- 20줄 이상의 old 블록을 정확히 매칭하려고 시도하기
- 섹션 번호를 대규모로 재정렬하기
```

### 4.2 replace-small-block

기존 문단을 바꿀 때는 1~5줄 단위로만 교체한다.

허용:

```text
- UPDATE 원칙 2~4줄 교체
- Required Checks에 bullet 2~3개 추가
```

금지:

```text
- 파일 절반 이상을 old/new로 교체
- 여러 섹션을 한 번의 거대한 StrReplaceFile에 넣기
```

### 4.3 json-field-edit

JSON은 필드 단위로만 수정한다.

- JSON 파싱 검증 필수
- 존재하지 않는 필드명 참조 금지
- 다음 Issue에서 생길 필드 미리 참조 금지

## 5. StrReplace 실패 처리: 하드 스톱

### 5.1 실패 횟수 규칙

같은 논리적 위치에서 `StrReplaceFile`이 실패하면 다음처럼 처리한다.

| 실패 횟수 | 행동 |
|---|---|
| 1회 | grep/Select-String으로 주변 5줄 확인 후, 더 짧은 앵커로 1회 재시도 |
| 2회 | 즉시 중단 보고 |

**중요:** 2회 실패 후 다음 행동은 금지한다.

```text
- “단순한 내 실수라서 한 번 더 시도”
- WriteFile로 전체 파일 재작성
- Python/PowerShell로 우회 수정
- 같은 위치를 다른 표현으로 계속 시도
```

### 5.2 중단 보고 형식

```md
## 중단 보고

- 실패 위치: `파일명`의 `섹션/앵커`
- 실패한 작업: 새 섹션 삽입 / 문단 교체 / JSON 필드 수정
- 현재까지 성공한 변경: ...
- 실패 원인 추정: 앵커 불일치 / 들여쓰기 차이 / 오타 / 코드블록 fence
- 필요한 다음 조치: 사용자가 현재 섹션 10줄을 제공하거나, 전체 재작성 승인을 해야 함
```

중단 보고 후 “완료”라고 쓰지 않는다.

## 6. 전체 WriteFile 금지

전체 파일 `WriteFile` 재작성은 기본 금지다.

허용되는 경우는 단 하나다.

```text
사용자가 명시적으로 “전체 파일 재작성해도 된다”고 승인한 경우
```

그 외에는 다음을 하지 않는다.

```text
- 이전 ReadFile 기억을 바탕으로 전체 파일 재구성
- 성공한 부분과 실패한 부분을 섞어 전체 덮어쓰기
- 원래 파일의 오타/문체/공백을 추측으로 보존하거나 수정
```

이유:

- 기존 변경사항을 덮어쓸 수 있다.
- 읽지 않은 최신 변경을 삭제할 수 있다.
- 오타를 새로 만들 수 있다.
- 토큰을 크게 낭비한다.

---

## 7. 한글 오타/인코딩 처리 강화

### 7.1 “폴드/폴더”류 오타 처리

한글 한 단어가 의심되면 전체 파일을 다시 쓰지 않는다.

절차:

1. `grep -n -C 1 "의심 단어" 파일`로 해당 줄만 확인한다.
2. 실제 Issue 요구사항과 관련된 오타인지 확인한다.
3. Issue 범위 밖이면 수정하지 않고 “기존 오타로 보임”이라고만 보고한다.
4. Issue 범위 안이면 해당 줄 하나만 교체한다.

금지:

```text
- 오타 원인을 장문으로 추측하기
- “원래 그랬는지” 혼란스러우면 전체 WriteFile하기
- 콘솔 출력 깨짐을 파일 오타로 단정하기
```

### 7.2 기존 오타와 신규 오타 구분

수정 전부터 있던 오타는 현재 Issue의 완료 기준이 아니면 고치지 않는다.

보고 예:

```text
참고: `폴드` 표기가 검색되었으나 현재 Issue 요구사항이 아니므로 수정하지 않음.
```

## 8. 복잡 Issue 처리 방식

Issue가 여러 파일을 건드리면 한 번에 모두 고치지 말고 **파일별 미니 패치**로 나눈다.

순서:

1. 가장 핵심 파일 1개 수정
2. 검색 검증
3. 다음 파일 수정
4. 검색 검증
5. 최종 전체 검증

금지:

```text
- 3~4개 파일에 대한 StrReplaceFile을 한꺼번에 병렬 실행
- 여러 파일 실패를 한 번에 분석하기
- 실패한 파일을 놔두고 완료 보고하기
```

## 9. Markdown 코드블록 삽입 규칙

중첩 코드블록이 있는 Markdown을 삽입할 때는 다음 중 하나만 사용한다.

### 9.1 틸다 fence 사용

````text
~~~markdown
### 예시
```text
내부 코드블록
```
~~~
````

### 9.2 코드블록 없이 예시를 bullet로 풀기

템플릿 예시가 짧으면 코드블록 대신 bullet로 쓴다.

금지:

```text
- 깨진 중첩 fence를 그대로 삽입하기
- 코드블록 닫힘 여부를 검증하지 않고 완료 보고하기
```

검증:

```bash
grep -n "~~~\|```" 파일
```

## 10. 검증 규칙

작업 후 최소 검증:

```text
1. 수정된 각 파일에서 핵심 문구 grep 확인
2. JSON 수정 시 JSON 파싱 검증
3. git status --short
4. 실패한 패치가 있었으면 완료 보고 금지
```

### 10.1 검색 확인 예시

```bash
grep -n -C 2 "PLAN 템플릿 종류" agent_rules/memory_management_guideline.md
grep -n -C 2 "Mini PLAN" agent_rules/input_guide.md
grep -n -C 2 "입력 커버리지 검증" agent_rules/validation_guide.md
```

## 11. 최종 보고 형식

성공 시:

```md
## Issue N 처리 결과

### 수정 파일
- `파일1`
- `파일2`

### 변경 요약
- 핵심 변경 1
- 핵심 변경 2
- 핵심 변경 3

### 검증
- 검색 확인: 통과
- JSON 파싱: 통과 / 해당 없음
- git status: `...`

### 미수정 파일
- `파일명`: 요구사항/완료 기준에 직접 변경 항목이 없어 미수정
```

중단 시:

```md
## Issue N 중단 보고

### 완료된 변경
- ...

### 실패한 변경
- ...

### 중단 사유
- 같은 위치 StrReplaceFile 2회 실패

### 다음 조치
- 현재 섹션 10줄 제공 또는 전체 재작성 승인 필요
```

금지:

```text
- 실패가 있었는데 완료 보고하기
- git status 없이 완료 보고하기
- 수정하지 않은 파일을 “검토 완료”라고 쓰기
```

## 12. 기존 problem_solving_role.md 대비 강화점

이 role은 기존 `problem_solving_role.md`보다 아래를 더 강하게 제한한다.

| 문제 | 새 제한 |
|---|---|
| 수정 대상 파일을 모두 읽음 | 구체적 변경 요구가 있는 파일만 읽기 |
| 큰 old/new 블록 매칭 실패 | append-before-heading 우선 |
| 실패 후 계속 우회 | 2회 실패 시 하드 스톱 |
| 전체 WriteFile로 덮어쓰기 | 사용자 승인 없이는 금지 |
| 장문 Thinking 출력 | Scope Card + 최종 보고만 허용 |
| 오타/인코딩 혼란 | 해당 줄만 확인, Issue 범위 밖이면 미수정 |
| 병렬 패치 후 다중 실패 | 파일별 미니 패치 순차 처리 |

## 13. Kimi/Codex에 붙일 실행 프롬프트

```text
이번 작업은 단일 Issue 처리다.
surgical_patch_role.md를 problem_solving_role.md보다 우선 적용한다.

제한:
- 구체적 변경 요구가 있는 파일만 읽는다.
- 수정 대상에 있어도 완료 기준에 직접 항목이 없으면 읽지 않는다.
- 전체 ReadFile은 수정 파일당 최대 1회, 총 3회까지다.
- 새 섹션 삽입은 큰 블록 교체가 아니라 append-before-heading 방식으로 한다.
- 20줄 이상의 old 블록 매칭을 금지한다.
- 같은 논리적 위치에서 StrReplaceFile이 2회 실패하면 즉시 중단 보고한다.
- 2회 실패 후 WriteFile/Python/Shell로 우회 수정하지 않는다.
- 전체 파일 WriteFile 재작성은 사용자 승인 없이는 금지한다.
- 여러 파일 패치는 병렬 실행하지 말고 파일별로 수정 후 검색 확인한다.
- 장문 Thinking, old/new 전문, Issue 본문 재요약을 출력하지 않는다.
- 최종 보고는 20줄 이내로 작성한다.

필수:
- 수정된 핵심 문구를 grep/Select-String으로 확인한다.
- JSON 수정 시 파싱 검증한다.
- git status --short를 보고한다.
- 실패한 패치가 하나라도 있으면 완료 보고하지 않는다.
```
