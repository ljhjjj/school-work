# 자율형 마크다운 강의노트 생성 알고리즘 사용 가이드

## 1. 목적

이 문서는 강의 자료(`.txt`, `.pdf`, `.jpg`, `.png`)를 입력받아 고품질 마크다운 강의노트로 변환하는 **자율형 마크다운 편집 알고리즘**의 사용 방법을 정리한다.

이 알고리즘은 단순 요약기가 아니라, 다음 세 가지를 동시에 관리하는 작업 시스템이다.

- **입력 자료의 범위 판단**: 어떤 자료를 본문 기준으로 삼을지 결정한다.
- **PLAN 기반 진행 관리**: 긴 작업을 차시 단위 PLAN 파일로 나누어 이어서 수행한다.
- **검증 및 보고**: 결과물의 형식, 수식, 잔여 이슈, 변경 로그를 검증하고 보고한다.

---

## 2. 핵심 개념

### 2.1 `primary_scope_anchor`

`primary_scope_anchor`는 본문에 포함할 **주제와 범위를 최종 결정하는 기준 자료**이다.

중요한 점은 다음과 같다.

- 표기 정확도 판단 우선순위와 본문 범위 우선순위는 다르다.
- 손글씨 PDF나 강의자료 PDF가 더 정확해 보이더라도, `primary_scope_anchor` 범위를 넘어서는 내용은 본문에 넣지 않는다.
- 범위가 불확실하면 본문에 단정적으로 포함하지 않고 PLAN이나 최종 리포트에 이슈로 남긴다.

### 2.2 PLAN 파일

PLAN 파일은 작업 기억 장치이다.

- 파일명 형식: `[과목명]/[과목명]_[주차]-[차시]_PLAN.md`
- 역할:
  - 작업 목표 기록
  - 입력 자료 해석 결과 기록
  - 작성 완료 섹션 체크
  - 재개 지점 표시
  - 판독 불가, 대응 불확실, 검증 이슈 기록

긴 문서 작업 중 토큰 한계, 시스템 오류, 판독 불가가 발생해도 PLAN 파일을 기준으로 이어서 작업할 수 있다.

### 2.3 입력 Manifest

`input_manifest.json`은 Codex가 어떤 작업을 처리해야 하는지 자동으로 발견하기 위한 작업 명세서이다.

- 기본 위치: `inbox/[과목명]/[주차]-[차시]/input_manifest.json`
- 역할:
  - 작업 상태 관리
  - 과목/주차/차시 지정
  - 입력 파일 목록 지정
  - 각 입력 파일의 역할 지정
  - 출력 노트와 PLAN 파일 경로 지정

## 4. 사용 절차 요약

### Step 1. 프로젝트 루트 준비

프로젝트 루트에 다음을 둔다.

- `AGENTS.md`
- `agent_rules/` 폴더
- `schemas/` 폴더
- `inbox/` 폴더

Codex는 모든 상대 경로를 프로젝트 루트 기준으로 해석해야 한다.

### Step 2. 입력 자료 배치

작업 단위 폴더를 만든다.

```text
inbox/[과목명]/[주차]-[차시]/
```

예시:

```text
inbox/고체역학/3-1/
```

그 안에 강의 자료를 넣는다.

```text
inbox/고체역학/3-1/lecture_stt.txt
inbox/고체역학/3-1/handwritten_note.pdf
inbox/고체역학/3-1/lecture_slide.pdf
```

### Step 3. `input_manifest.json` 작성

Manifest는 파일명 추론보다 우선한다.

```json
{
  "status": "pending",
  "task_priority": 10,
  "subject": "고체역학",
  "week": "3",
  "session": "1",
  "task_type": "new_note",
  "input_root": "inbox/고체역학/3-1",
  "output_note": "고체역학/고체역학 3-1.md",
  "plan_file": "고체역학/고체역학_3-1_PLAN.md",
  "sources": [
    {
      "path": "inbox/고체역학/3-1/lecture_stt.txt",
      "type": "stt_txt",
      "role": "primary_scope_anchor",
      "priority": 1
    },
    {
      "path": "inbox/고체역학/3-1/handwritten_note.pdf",
      "type": "note_pdf",
      "role": "bounded_merge_source",
      "priority": 2
    },
    {
      "path": "inbox/고체역학/3-1/lecture_slide.pdf",
      "type": "ppt_pdf",
      "role": "verification_only",
      "priority": 3
    }
  ]
}
```

### Step 4. Codex 실행 요청

Codex에는 짧고 명확하게 요청한다.

```text
AGENTS.md와 agent_rules를 따라 inbox의 pending 작업을 처리해줘.
마크다운 강의노트를 생성하고 PLAN 및 검증 리포트를 업데이트해줘.
```

특정 차시만 처리하려면 다음처럼 요청한다.

```text
AGENTS.md와 agent_rules를 따라 inbox/고체역학/3-1 작업만 처리해줘.
```

### Step 5. 결과 확인

작업 완료 후 다음 파일을 확인한다.

```text
[과목명]/[과목명] [주차]-[차시].md
[과목명]/[과목명]_[주차]-[차시]_PLAN.md
```

최종 보고에서는 최소한 다음이 구분되어야 한다.

- 완료한 작업
- 생성 또는 수정한 파일
- 검증 결과
- 자동 수정 내역
- 건너뛴 내용
- 잔여 이슈
- 다음 재개 지점

## 5. 입력 조합별 처리 원칙

### 5.1 순수 손글씨 PDF + 음성 TXT + 강의자료 PDF

| 자료 | 역할 |
|---|---|
| 음성 TXT | 본문 범위 기준 자료 |
| 손글씨 PDF | TXT 범위 안에서 판서 수식, 그래프, 예제 풀이 복원 |
| 강의자료 PDF | 용어, 공식, 기호, 목차 후보 검증 |

처리 원칙은 다음과 같다.

- 본문 범위의 최종 기준은 음성 TXT이다.
- TXT에서 실제로 설명된 내용만 본문에 포함한다.
- TXT에 “이 식”, “이 그래프”, “여기”, “정리하면” 같은 지시어가 있으면 대응되는 손글씨 판서는 TXT 범위 안으로 본다.
- 손글씨에만 보이는 내용은 TXT와 대응되지 않으면 본문에 넣지 않는다.
- 강의자료 PDF에만 있는 내용은 `[자료 PDF 초과 범위]`로 기록하고 본문에는 넣지 않는다.

### 5.2 필기 PDF + 강의자료 PDF

| 자료 | 역할 |
|---|---|
| 필기 PDF | 본문 범위 기준 자료 |
| 강의자료 PDF | 공식 표기, 영문 용어, 그림, OCR/필기 오타 보정 |

처리 원칙은 다음과 같다.

- 필기 PDF에 있는 주제, 수식, 설명, 예제만 본문에 포함한다.
- 강의자료 PDF가 더 넓어도 필기 PDF와 직접 대응되는 부분만 사용한다.
- 필기 PDF에만 있는 교수 설명, 직관적 해석, 시험 강조점은 필기 PDF를 우선한다.

### 5.3 자료 PDF 위에 필기만 있는 경우

| 자료 | 역할 |
|---|---|
| 자료 PDF | 슬라이드 제목, 공식, 예제 번호, 그림 순서 기준 |
| 손글씨 필기 | 해당 위치의 보충 정보 |

처리 원칙은 다음과 같다.

- 자료 PDF의 구조를 본문 구조의 기준으로 사용한다.
- 손글씨는 `교수 보충`, `주의`, `계산 힌트`, `강조`, `풀이 근거`로 분류한다.
- 필기 없는 슬라이드는 과도하게 확장하지 않고 기본 요약만 작성한다.
- 풀이가 없는 문제는 새로 창작하지 않고 `자료에 풀이 없음`으로 기록한다.

## 7. 마크다운 노트 작성 규칙

### 7.1 파일명과 저장 경로

```text
[과목명]/[과목명] [주차]-[차시].md
```

예시:

```text
고체역학/고체역학 3-1.md
```

### 7.2 제목 구조

```markdown
# 고체역학 3-1: 응력과 변형률

## 1. 응력(Stress)의 개념

본문 내용

---

## 2. 변형률(Strain)의 정의

본문 내용
```

### 7.3 문체

- 기본 언어는 한국어이다.
- 문어체 `~한다`, `~이다`를 사용한다.
- 기술 용어는 한국어를 먼저 쓰고 필요하면 영어를 병기한다.

예시:

```markdown
- **전단응력(Shear Stress):** 단면에 평행하게 작용하는 응력이다.
```

### 7.4 수식 규칙

인라인 수식은 다음처럼 쓴다.

```markdown
응력은 $\sigma$로 나타낸다.
```

블록 수식은 앞뒤 줄바꿈을 포함해 독립 블록으로 쓴다.

```markdown
$$
\sigma = \frac{P}{A}
$$
```

### 7.5 내용 유형별 포맷

| 내용 유형 | 권장 포맷 |
|---|---|
| 개념 정의 | 불릿 포인트 |
| 특징 정리 | 불릿 포인트 |
| 수식 도출 | 단계별 번호 또는 상세 서술 |
| 논리 전개 | 줄글 또는 번호 단계 |
| 예시 | `> **예시:**` 블록 |
| 핵심 강조 | `> **핵심:**` 블록 |

## 8. PLAN 파일 운영 규칙

PLAN 파일은 차시 단위로 분리한다.

```text
[과목명]/[과목명]_[주차]-[차시]_PLAN.md
```

예시:

```text
고체역학/고체역학_3-1_PLAN.md
```

### 8.1 PLAN 기본 구조

```markdown
### 📋 목표: 고체역학 3주차 1차시 (응력과 변형률)
- **타겟 출력 파일:** `고체역학/고체역학 3-1.md`

#### Phase 1: 원본 데이터 분석 및 목차 설계
- [x] 1단계: 입력 파일 확인
- [x] 2단계: primary_scope_anchor 확정

#### Phase 2: 본문 마크다운 변환
- [x] 1. 응력 개념 작성
- [ ] 2. 변형률 정의 작성 👈 **현재 포커스 / 재개 지점**

#### ⚠️ 실행 이슈
- 없음

#### Phase 3: 최종 검수 및 보고
- [ ] 1단계: 마크다운 구조 검증
- [ ] 2단계: 수식 렌더링 검증
- [ ] 3단계: 최종 리포트 출력
```

### 8.2 체크박스 업데이트

- H2 또는 H3 단위 작성이 끝나면 즉시 `[x]`로 변경한다.
- 중간에 멈추면 현재 재개 지점을 명확히 표시한다.
- 불확실한 내용은 본문에 억지로 넣지 않고 PLAN에 이슈로 남긴다.

## 9. 불확실성 태그 사용법

### 9.1 정상 판정/범위 관리 태그

이 태그들은 validation warning으로 보지 않는다.

| 태그 | 의미 |
|---|---|
| `[확정]` | 두 개 이상의 원본 자료가 일치함 |
| `[보정]` | 필기/OCR/STT가 불명확하지만 자료로 확인됨 |
| `[필기 우선]` | 강의자료에는 없지만 필기 PDF에 명확히 있음 |
| `[TXT 우선]` | 강의자료에는 있으나 TXT에서 설명되지 않아 제외함 |
| `[자료 PDF 초과 범위]` | 자료 PDF에는 있으나 현재 본문 범위를 넘어감 |
| `[제외]` | 현재 입력 조합의 범위 기준 밖이라 제외함 |

### 9.2 잔여 이슈 태그

이 태그들은 최종 리포트의 `[잔여 이슈]`에 포함한다.

| 태그 | 의미 |
|---|---|
| `[?]` | 판독 불가 또는 확정 불가 |
| `[필기 판독 불확실]` | 손글씨가 작거나 흐려 확정할 수 없음 |
| `[TXT 누락 의심]` | STT가 손글씨 풀이 구간을 누락한 것으로 보임 |
| `[대응 불확실]` | TXT와 손글씨의 대응 근거가 약함 |
| `[확인 필요]` | 자동 확정이 어려움 |
| `[수동 입력 필요]` | 자동 판독이 불가능함 |

## 10. 웹 교차검증 사용 원칙

웹 교차검증은 본문 확장용이 아니다.

허용되는 사용 범위는 다음으로 제한한다.

- STT 오류가 의심되는 전문 용어 확인
- 보편 공식, 단위, 수학/물리 기호 오타 확인
- 영문/한글 표준 전공 용어 표기 확인

금지되는 사용은 다음과 같다.

- 웹 자료를 근거로 강의 내용을 확장하는 것
- 블로그나 Q&A 하나만 보고 본문을 확정하는 것
- 원본 강의자료의 전개 방식이나 예제 순서를 임의 변경하는 것
- 검색 결과가 불확실한데 단정적으로 수정하는 것

웹 검증을 통해 수정하거나 추정한 내용은 최종 리포트와 PLAN에 남긴다.

## 11. 검증 체크리스트

작업 완료 전 다음을 확인한다.

```markdown
## ✅ Validation Checklist

### 파일 및 경로
- [ ] 노트 파일 경로가 `[과목명]/[과목명] [주차]-[차시].md` 형식이다.
- [ ] PLAN 파일 경로가 `[과목명]/[과목명]_[주차]-[차시]_PLAN.md` 형식이다.
- [ ] 모든 `.md` 파일이 UTF-8로 저장되었다.

### 마크다운 구조
- [ ] H1이 정확히 1개이다.
- [ ] H1 형식이 `# [과목명] [주차]-[차시]: [주제]`이다.
- [ ] H2 형식이 `## 1. 주제명` 형태이다.
- [ ] 주요 H2 섹션 뒤에 구분선 `---`이 있다.

### 수식
- [ ] 인라인 수식은 `$...$` 형식이다.
- [ ] 블록 수식은 독립된 `$$` 블록으로 작성되었다.
- [ ] 한 줄에 `$$ ... $$`를 쓰지 않았다.
- [ ] `<math>` 태그를 사용하지 않았다.

### 범위 관리
- [ ] 본문 내용이 `primary_scope_anchor` 범위를 넘지 않는다.
- [ ] 초과 범위는 본문이 아니라 PLAN 또는 리포트에 기록되었다.
- [ ] 불확실한 내용은 잔여 이슈 태그로 남겼다.

### PLAN 및 보고
- [ ] 완료된 섹션의 PLAN 체크박스가 `[x]`로 갱신되었다.
- [ ] 재개 지점이 명확하다.
- [ ] 자동 수정 내역이 기록되었다.
- [ ] 잔여 이슈가 최종 리포트에 포함되었다.
```

## 12. 안전 규칙

### 12.1 파일 보호

- 사용자 파일을 덮어쓰기 전에 현재 내용을 확인한다.
- 내가 수정하지 않은 diff는 임의로 건드리지 않는다.
- 요청받은 과목/주차/차시에 해당하는 파일만 수정한다.

### 12.2 명령 실행

위험 명령은 사용자 명시 승인 없이 실행하지 않는다.

예시:

```text
rm -rf
Remove-Item
git reset
git clean
git checkout
git restore
git push
git commit
npm install
pip install
```

### 12.3 패키지 설치

검증 도구가 없다는 이유로 패키지를 임의 설치하지 않는다.
가능하면 수동 정규식 검사나 기존 스크립트를 사용한다.

## 14. 권장 실행 프롬프트

### 14.1 전체 pending 작업 처리

```text
AGENTS.md와 agent_rules를 따라 inbox의 pending 작업을 처리해줘.
작업 전후 git 상태를 확인하고, PLAN 파일과 마크다운 노트를 생성/수정한 뒤 validation_guide 기준으로 검증해줘.
최종 보고에는 완료 작업, 건너뛴 범위, 잔여 이슈, 다음 재개 지점을 구분해줘.
```

AGENTS.md와 agent_rules를 따라 고체역학/고체역학_3-1_PLAN.md의 재개 지점부터 이어서 작성해줘.
